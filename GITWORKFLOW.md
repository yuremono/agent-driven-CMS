# Git Workflow

- 破壊的な git 操作は使いません。

## ブランチ運用

1. `main` から feature ブランチを切ります。
2. 必要なら同じ feature 系ブランチの worktree を複数作って並行作業します。
3. 1 ブランチは 1 テーマに絞ります。
4. 実装が進んだら、早めに remote へ push して共有可能な状態にします。
5. 作業がまとまったら、`main` 向けに Pull Request を作ります。
6. レビュー指摘があれば、同じブランチに追加 commit を積んで push し直します。
7. 承認後に merge します。
8. merge 後は、必要に応じてローカルの `main` を更新します。

## コミットの考え方

- コミットメッセージは `type: description` を基本にします。
- 例: `feat: add bridge session provider`
- 使う type は、`feat` / `fix` / `refactor` / `docs` / `test` / `chore` を中心にします。

## Pull Request の作り方

- PR は `main` 向けに作ります。
- 差分確認は `git diff main...HEAD` を基準にします。
- PR の本文には、以下を含めます。
  - 何を直したか
  - どこが変わったか
  - どう確認したか
  - 残っている注意点

## このプロジェクトでの実務ルール

- UI、bridge、テスト、ドキュメントは、必要に応じて worktree で分けて進めます。
- `app/api/*` の変更は Node runtime 前提で扱います。
- 認証情報や秘密情報はブラウザに流しません。
- 開発用 UI は `development` 時だけ表示します。
- `admin` は検証用の画面として扱いますが、開発時のチャットインプットは全ページで共通にします。

## テスト

- bridge、状態管理、認証、SSE、表示条件を変えたときはテストを確認します。
- 軽い文言修正だけなら、無理に build/test を回しません。
- UI の見た目や表示条件を変えたときは、ブラウザでの確認を優先します。

## このリポジトリの優先順位

1. 安全性
2. 既存の挙動を壊さないこと
3. 差分を小さく保つこと
4. レビューしやすいこと
5. 必要なときだけ広く調べること
